**Why re-export shims instead of deletion:**
Research confirmed 13+ components outside Phase 2 scope (Navbar, EventCard, ProtectedRoute, MobileNavigation, BottomNav, FriendRequestsMenu, NotificationsDropdown, NotificationsPage, EmailVerificationBanner, ShareMemoryDialog, UserProfileDialog, MatchResultsDialog, RegisterDialog) still import from `AuthContext.tsx` or `useAuth.ts`. Deleting these files would break the build. Re-exporting satisfies the requirement (Firebase auth code is gone) while maintaining backward compatibility.

**Step 1: Replace `src/contexts/AuthContext.tsx` entirely with:**
```typescript
// Legacy AuthContext — re-exports from SupabaseAuthContext for backward compatibility.
// All Firebase auth code has been removed. Components importing from this path
// now transparently use Supabase auth.
// TODO: Update remaining importers to use SupabaseAuthContext directly, then delete this file.
export { SupabaseAuthProvider as AuthProvider, useAuth } from './SupabaseAuthContext';
```

This file should contain ONLY the re-export above — no imports from firebase/auth, no createContext, no Firebase User type. The `AuthProvider` name is preserved as an alias of `SupabaseAuthProvider` for any components that import `AuthProvider`.

**Step 2: Replace `src/hooks/useAuth.ts` entirely with:**
```typescript
// Legacy useAuth hook — re-exports from SupabaseAuthContext for backward compatibility.
// All Firebase auth code has been removed.
// TODO: Update remaining importers to use SupabaseAuthContext directly, then delete this file.
export { useAuth } from '../contexts/SupabaseAuthContext';
```

This file should contain ONLY the re-export above — no Firebase SDK imports, no auth state management, no onAuthStateChanged.

**CRITICAL WARNING:** After this change, ALL components using `useAuth()` (from either import path) will receive a Supabase `User` object. Any component accessing `user.uid` (Firebase) will get `undefined` — they need `user.id` (Supabase). However, fixing those 13+ components is OUT OF SCOPE for this phase. The re-export ensures the import paths work; the property name difference is a known issue that will surface as a TypeScript error in those files and must be addressed in a future milestone.
